Monitoramento & Observabilidade

Guia prático para entender como funciona uma stack completa de observabilidade usando containers Docker.

Prometheus Grafana Loki Promtail cAdvisor Node Exporter Nginx

🔧 As Ferramentas da Stack

🔥
Prometheus
localhost:9090

Banco de dados de séries temporais (time-series). A cada 15 segundos, ele "raspa" (scrape) os endpoints /metrics de cada serviço e armazena os dados localmente. Usa a linguagem PromQL para consultas.

métricas pull-based
↗ Abrir Prometheus
📊
Grafana
localhost:3000 · admin / admin

Plataforma de visualização e dashboards. Conecta em múltiplas fontes de dados (Prometheus, Loki, etc.) e exibe gráficos, alertas e painéis interativos em tempo real. É o "painel de controle" da stack.

dashboards alertas
↗ Abrir Grafana
📋
Loki
localhost:3100

Sistema de agregação de logs — funciona como um "Prometheus para logs". Não indexa o conteúdo dos logs (o que o torna eficiente), apenas os metadados (labels). Usa a linguagem LogQL para consultas.

logs labels
↗ Ver Logs no Grafana
📡
Promtail
agente interno

Agente que roda nos hosts e coleta os arquivos de log dos containers Docker (/var/lib/docker/containers). Adiciona labels (nome do container, stream, etc.) e envia para o Loki via HTTP push.

agente push-based
📦
cAdvisor
localhost:8081

Container Advisor — criado pelo Google. Monitora em tempo real o consumo de CPU, memória, rede e disco de cada container em execução. Expõe endpoint /metrics que o Prometheus coleta.

containers Google OSS
↗ Abrir cAdvisor
🖥️
Node Exporter
:9100 (interno)

Exporta métricas do sistema operacional host: CPU total, memória disponível, espaço em disco, tráfego de rede, carga do sistema (load average), etc. Indispensável para monitorar a saúde da máquina.

host OS Linux/Mac
💡 cAdvisor vs Node Exporter: cAdvisor monitora cada container individualmente. Node Exporter monitora o servidor como um todo. Você precisa dos dois para ter visibilidade completa.
💡 Loki vs Elasticsearch: O Loki foi projetado para ser muito mais simples e barato. Não indexa o texto dos logs — apenas os metadados (labels). Para buscas full-text complexas, o Elasticsearch é mais poderoso, mas muito mais pesado.

O Prometheus usa modelo pull (ele busca as métricas). O Loki usa modelo push (o Promtail envia os logs).

📋 Passo a Passo — Explorando a Stack

1

Confirmar que todos os containers estão rodando

Antes de tudo, verifique se todos os serviços subiram corretamente. Todos devem estar como Up.

docker compose ps
# Esperado: prometheus, grafana, loki, promtail, cadvisor, node-exporter, nginx — todos Up
✅ Se algum estiver como Exit, rode docker compose logs <nome> para ver o erro.
2

Acessar o Prometheus e verificar os targets

O Prometheus mostra quais serviços ele está monitorando. Todos devem estar verdes (UP).

# Abra no navegador:
http://localhost:9090/targets

Você verá três targets: prometheus, cadvisor e node-exporter. Estado UP = coletando com sucesso.

💡 Cada target expõe um endpoint /metrics com centenas de métricas em texto plano. Tente acessar: http://localhost:9090/metrics
3

Fazer sua primeira query PromQL no Prometheus

Acesse a aba Graph do Prometheus e execute uma query para ver dados ao vivo.

# Abra:
http://localhost:9090/graph

# Cole esta query e clique em Execute:
container_memory_usage_bytes{name!=""}

Você verá um gráfico com o uso de memória de cada container. Alterne entre Table e Graph para ver os dois formatos.

4

Acessar o Grafana e explorar os datasources

O Grafana já vem configurado automaticamente com Prometheus e Loki como fontes de dados.

http://localhost:3000
# Login: admin / admin  →  clique Skip ao pedir troca de senha

# Para ver os datasources:
# Menu lateral → Connections → Data sources
✅ Você verá Prometheus (marcado como padrão) e Loki já configurados sem nenhuma intervenção manual. Isso é o provisioning.
5

Visualizar logs no Grafana com Loki (Explore)

O Explore é onde você faz queries ao vivo sem precisar criar um dashboard.

# Acesse:
http://localhost:3000/explore

# No seletor de datasource (topo), escolha: Loki
# No campo de query, digite:
{job="docker"}

# Clique em "Run query" ou Shift+Enter

Você verá todos os logs de todos os containers em tempo real. Clique em Live para acompanhar as entradas chegando.

6

Gerar tráfego para ver métricas e logs mudarem

Faça requisições ao nginx para gerar logs e veja eles aparecendo no Loki.

# Terminal (Linux/Mac):
for i in $(seq 1 50); do curl -s http://localhost:8080 > /dev/null; done

# Gerar erros 404:
for i in $(seq 1 20); do curl -s http://localhost:8080/nao-existe; done

# Windows PowerShell:
1..50 | ForEach-Object { Invoke-WebRequest http://localhost:8080 -UseBasicParsing | Out-Null }
⚡ Após as requisições, volte ao Grafana Explore com a query {tag="nginx-app"} e veja os logs de acesso chegando.
7

Simular carga de CPU

Suba um container temporário que consuma CPU e observe o gráfico do host subir no Prometheus via Node Exporter.

# Cria carga em 2 CPUs por 45 segundos:
docker run --rm --name stress-test alpine \
  sh -c "cat /dev/urandom | gzip -9 > /dev/null &  cat /dev/urandom | gzip -9 > /dev/null; sleep 45"

# Query que vai mostrar o impacto (Node Exporter — funciona no Docker Desktop):
100 - (avg by(instance)(rate(node_cpu_seconds_total{mode="idle"}[30s])) * 100)

# ⚠️ A query abaixo NÃO funciona no Docker Desktop (só Linux nativo):
# rate(container_cpu_usage_seconds_total{name="stress-test"}[30s]) * 100
💡 No Docker Desktop, use o Node Exporter para ver o impacto de CPU — a métrica node_cpu_seconds_total reflete o consumo real do host WSL2.

Verificar o cAdvisor diretamente

O cAdvisor tem sua própria interface web com visualização por container.

🔍 Queries de Referência

⚠️ Docker Desktop (Windows/Mac): queries de container individuais (marcadas ⚠️) só funcionam em Linux nativo — o cAdvisor não enumera containers dentro da VM WSL2. Todas as queries marcadas com ✅ funcionam normalmente.
💡 Essas queries funcionam tanto no Prometheus (:9090/graph) quanto no Grafana Explore selecionando o datasource Prometheus.

✅ ❤️ Status dos targets — quais serviços estão sendo coletados

Retorna 1 (UP) ou 0 (DOWN) para cada target. É a primeira query a executar para saber se a stack está saudável.

up

✅ 🖥️ CPU do host em % — uso total excluindo o modo idle

Calcula a taxa de variação do tempo de CPU em modo idle nos últimos 5 min e subtrai de 100%. avg by(instance) agrega todos os núcleos em um único valor.

100 - (avg by(instance)(rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

✅ 🖥️ CPU por modo (user, system, iowait, steal...)

Mostra quanto tempo a CPU passa em cada modo. iowait alto indica gargalo de disco; steal alto indica contenção em VM compartilhada.

rate(node_cpu_seconds_total[5m])

✅ 💾 Memória RAM usada no host (bytes)

Total instalado menos o disponível. Mais preciso que MemFree — considera buffers e cache que o kernel pode liberar se necessário.

node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes

✅ 💾 Uso de memória em % do total

Versão percentual — mais fácil de interpretar num gauge do Grafana. Subtrai o disponível do total e divide. Resultado entre 0 e 100.

(1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100

✅ 💿 Espaço livre em disco no filesystem raiz (%)

Divide o espaço disponível pelo total no ponto de montagem /. Útil para criar alertas de disco cheio antes de atingir 100%.

(node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"}) * 100

✅ 📶 Tráfego de rede recebido (bytes/s) — todas as interfaces

Taxa de bytes recebidos nos últimos 5 minutos. Sem filtro de interface, mostra todas. Troque [5m] por [1m] para variações mais rápidas.

rate(node_network_receive_bytes_total[5m])

✅ 📡 Tráfego de rede enviado (bytes/s) — todas as interfaces

Mesma lógica, mas para o tráfego de saída (transmit). Combine com a query de recebimento num painel para ver upload e download lado a lado.

rate(node_network_transmit_bytes_total[5m])

✅ 🔄 Load average — carga do sistema (1, 5 e 15 minutos)

Média de processos aguardando CPU. Valores acima do número de núcleos indicam sobrecarga. Compare os três para ver a tendência: se load15 > load1, a carga está diminuindo.

# 1 minuto (mais instável, reage rápido a picos)
node_load1

# 5 minutos (referência mais usada)
node_load5

# 15 minutos (tendência de longo prazo)
node_load15

✅ ⏱️ Uptime do host em horas

Diferença entre o tempo atual e o tempo de boot, convertida para horas. Útil para confirmar se houve reinicialização inesperada.

(node_time_seconds - node_boot_time_seconds) / 3600

✅ 🌡️ Tempo de coleta de cada target (scrape duration)

Quanto tempo o Prometheus levou para coletar métricas de cada target. Picos indicam lentidão no exporter — se ultrapassar 15s, o target pode ficar DOWN.

scrape_duration_seconds

✅ 📊 Total de séries temporais armazenadas (cardinalidade)

Indica quantas séries únicas o Prometheus mantém em memória. Muitas séries consomem mais RAM e CPU. Útil para monitorar a saúde do próprio Prometheus.

prometheus_tsdb_head_series

⚠️ 📦 Memória usada por container em bytes — só Linux nativo

Retorna o uso de RAM de cada container via cAdvisor. No Docker Desktop (Windows/Mac), o cAdvisor não acessa os cgroups dentro da VM WSL2 e a query retorna vazia.

# Não retorna resultados no Docker Desktop/Windows
container_memory_usage_bytes{name!=""}

⚠️ 🔥 CPU por container em % — só Linux nativo

Taxa de uso de CPU de cada container nos últimos 5 minutos. Requer acesso direto aos cgroups do host — disponível apenas em Linux nativo, não no Docker Desktop.

# Não retorna resultados no Docker Desktop/Windows
rate(container_cpu_usage_seconds_total{name!=""}[5m]) * 100

⚠️ 🔢 Número de containers em execução — só Linux nativo

Conta quantos containers únicos estão ativos segundo o cAdvisor. Útil para dashboards de infraestrutura. Não funciona no Docker Desktop.

# Não retorna resultados no Docker Desktop/Windows
count(container_last_seen{name!=""})
✅ Todas as queries LogQL abaixo funcionam no Docker Desktop. Acesse pelo Grafana Explore: localhost:3000/explore → selecione Loki como datasource.

📋 Todos os logs de todos os containers Docker

O seletor {job="docker"} é o ponto de entrada. Use como base antes de refinar com filtros adicionais.

{job="docker"}

🌐 Logs apenas do nginx (logs de acesso HTTP)

Filtra pela label tag que o Promtail extrai do nome do container. Você vê: IP, método HTTP, path, código de status e bytes transferidos.

{job="docker", tag="nginx-app"}

🤖 Logs do simulator (nosso app de testes)

Filtra os logs do container Python que gera eventos de INFO, WARN e ERROR. Use os botões do painel de testes e observe os logs chegando aqui em tempo real.

{tag="simulator"}

🔴 Filtrar linhas que contêm "ERROR" (exato, case sensitive)

O operador |= faz busca de substring no texto do log. Rápido e eficiente para filtros simples. Encadeie vários |= para filtros AND.

{job="docker"} |= "ERROR"

🔎 Filtrar por "error" ignorando maiúsculas/minúsculas (regex)

O operador |~ usa expressão regular. A flag (?i) torna a busca case-insensitive — captura "Error", "ERROR" e "error" ao mesmo tempo.

{job="docker"} |~ "(?i)error"

🚫 Excluir linhas que contenham determinado texto

O operador != é o inverso do |= — descarta linhas que contenham o texto. Útil para remover ruído, como health checks repetitivos dos logs.

{job="docker"} != "health"

🟠 Filtrar pelo stream stderr (onde erros costumam aparecer)

O Docker separa os logs em stdout e stderr. Aplicações costumam escrever mensagens de erro no stderr. Esta query isola apenas esse stream.

{job="docker", stream="stderr"}

🚨 Erros 404 do nginx

Combina filtro de container (nginx-app) com filtro de texto (404). Encadeie quantos filtros precisar — cada operador adicional refina ainda mais o resultado.

{tag="nginx-app"} |= "404"

📊 Contar linhas de log por container (últimos 5 min)

Agrega a contagem de linhas pela label tag. Mostra qual container está gerando mais logs — ótimo para detectar barulho excessivo de logging.

sum by(tag) (count_over_time({job="docker"}[5m]))

📈 Taxa de logs por segundo por container (logs/s)

Usando rate() igual ao PromQL, mas sobre linhas de log. Útil para detectar picos de logging e criar alertas de anomalia.

sum by(tag) (rate({job="docker"}[1m]))

🔢 Contar só os logs de ERROR nos últimos 10 minutos

Combina filtro de texto com agregação temporal. O resultado é um número escalar — ideal para um painel "stat" ou para criar um alerta automático no Grafana.

sum(count_over_time({job="docker"} |= "ERROR" [10m]))

🔍 Extrair código de status HTTP do nginx e filtrar erros 4xx/5xx

O parser pattern extrai campos nomeados do texto do log. Depois filtra só os status ≥ 400. Muito poderoso para análise de tráfego HTTP sem precisar de outro exporter.

{tag="nginx-app"} | pattern `<_> "<method> <path> <_>" <status> <_>` | status >= 400

📝 Logs do sistema operacional host

O Promtail coleta também os logs do sistema em /var/log do WSL2. Útil para ver eventos do kernel, autenticação SSH e serviços do host.

{job="varlogs"}

Crie dashboards com qualquer combinação de métricas e logs. Cada visualização é um "panel" que usa PromQL ou LogQL como fonte de dados.

# 1. Dashboards → New → New Dashboard
# 2. Clique em "+ Add visualization"
# 3. Selecione o datasource (Prometheus ou Loki)
# 4. Digite a query (PromQL ou LogQL)
# 5. Escolha o tipo de visualização (veja abaixo)
# 6. Clique Apply → salve o dashboard (ícone de disquete)

🎨 Tipos de visualização disponíveis no Grafana

Cada tipo é ideal para um caso de uso específico. No editor do panel, use o seletor no canto superior direito para trocar entre os tipos.

Time series → séries temporais (CPU%, rede, latência ao longo do tempo)
Gauge       → valor atual com limite máximo (uso de disco %, memória %)
Stat        → número único em destaque (uptime, total de erros)
Bar gauge   → comparar valores entre containers ou instâncias
Logs        → stream de logs do Loki (visualização nativa de log)
Table       → dados tabulares com múltiplas métricas em colunas
Pie chart   → proporções (% de memória por container)

⏱️ Ajustar o intervalo de tempo e auto-refresh

O seletor de tempo no canto superior direito de qualquer dashboard ou Explore controla o período exibido. Ative o auto-refresh para atualizações automáticas.

Last 5m   → ver picos em tempo real logo após um teste
Last 15m  → referência padrão durante a aula
Last 1h   → ver tendência de crescimento
Last 6h   → análise pós-incidente

# Auto-refresh: clique no seletor de tempo → escolha 5s, 10s ou 30s
# Tecla F5 ou ícone ↻ para atualizar manualmente

📖 Glossário Rápido

Observabilidade
Capacidade de entender o estado interno de um sistema a partir de suas saídas externas: métricas, logs e traces.
Métrica
Valor numérico medido ao longo do tempo. Ex: CPU%, bytes de memória, requisições/segundo.
Time Series
Sequência de valores numéricos indexados por tempo. É como o Prometheus armazena as métricas.
Scrape
Ação do Prometheus de "raspar" (requisitar) o endpoint /metrics de um serviço para coletar métricas.
Label
Par chave=valor que identifica uma série temporal ou entrada de log. Ex: container="nginx".
PromQL
Linguagem de consulta do Prometheus. Permite calcular taxas, médias e agregações sobre as métricas.
LogQL
Linguagem de consulta do Loki. Baseada em PromQL, mas para filtrar e agregar linhas de log.
Exporter
Serviço que expõe métricas no formato Prometheus de um sistema que não as expõe nativamente. Ex: Node Exporter, cAdvisor.
Provisioning
Configuração automática de datasources e dashboards no Grafana via arquivos YAML, sem intervenção manual.
cgroups
Recurso do kernel Linux que isola e limita CPU, memória e I/O de processos. Base do Docker e do cAdvisor.
Pull vs Push
Prometheus vai buscar as métricas (pull). Promtail envia os logs ao Loki (push). Ambos os modelos coexistem na stack.
Cardinality
Quantidade de séries temporais únicas. Alta cardinalidade (muitas labels únicas) impacta performance do Prometheus e Loki.
